# Lessons Learned（修正パターン記録）

## 1. 元のHTML/デザインを勝手に変更しない
- **状況**: WordPressテーマ化の際にナビを日本語化、セクション追加、ボタンテキスト変更など余計な変更を加えた
- **結果**: ユーザーが期待するデザインと異なるものになった
- **ルール**: 元のHTML/Figmaデザインを忠実に再現する。変更が必要な場合は事前に確認する

## 2. プリセットのCSS変数を勝手に変更しない
- **状況**: 元のHTMLがTailwind CSSを使用していたのに、独自CSSに書き直した
- **結果**: デザインが完全に崩れた
- **ルール**: プリセットで定義されたCSS変数（--bg/--accent等）の値を勝手に変更しない。変更が必要な場合は事前に確認する

## 3. wp_enqueue_scriptでTailwind CDNを読み込まない
- **状況**: WordPressの`wp_enqueue_script`でTailwind CDNを読み込んだ
- **結果**: configスクリプトとCDNの読み込み順序が保証されず、カスタムカラーが効かない
- **ルール**: Tailwind CDNとconfigは`header.php`に直接記述する（WP案件用）

## 4. GitHubが最新版とは限らない
- **状況**: GitHubのHTMLを最新版と思い込んだ
- **結果**: localhostで動いている版と比較して確認が必要だった
- **ルール**: 最新版がどこにあるか必ずユーザーに確認する

## 5. 量産時はスキル＋プリセットで品質担保
- **状況**: サイトを量産する際、毎回ゼロからUI設計するとクオリティにバラつきが出る
- **結果**: UI品質の不安が制作スピードのボトルネックになる
- **ルール**: スキルの参照ファイル＋カラープリセットを使用して品質を安定させる。業態別スキル（lp-salon/lp-barber等）で構造を統一し、プリセットで差別化する
