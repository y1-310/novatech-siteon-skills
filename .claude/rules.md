## 3. HTML品質（8項目）

10. セマンティックHTMLを使用する（header/nav/main/section/footer）
11. 全imgタグにalt属性を設定する（日本語）
12. 全imgタグにwidth・height属性を設定する
13. loading="lazy" を設定する（ヒーロー画像を除く）
14. lang="ja" をhtml要素に設定する
14a. `<meta name="color-scheme" content="light">` を必ず設定する（Chrome強制ダークモード対策）
15. 英語要素には lang="en" を個別指定する
16. フォームのlabelとinputを紐付ける
17. noscript対応を実装する（フェードイン→常時表示 / ナビ→フル表示 / マーキー→静止 / タブ→全展開）

## 4. CSS品質（13項目）

18. CSS変数を統一使用する（--bg / --bg-alt / --bg-dark / --text / --text-mid / --text-light / --accent / --line / --r）
19. CSS変数の値をハードコードしない
20. **禁止フォント：Inter / Arial / Roboto / Helvetica は使用禁止**
21. アニメーションは transform と opacity のみ使用する（top/left/marginのアニメーション禁止）
22. セクション間padding: 120px（desktop）/ 72px（mobile）
23. border-radius は var(--r) で統一する
24. 横線で区切る — セクション間に 1px solid var(--line)
25. 彩度を抑える — ビビッドな色は使わない
26. prefers-reduced-motion に対応する
27. !important は使用禁止（noscript内を除く）
28. 日本語フォントの全ウェイト読み込み禁止（最大3ウェイト：300/400/500）
29. CSS/JSの外部ファイル分離禁止（スタンダード時）
31. word-break: keep-all を適用する要素は、親コンテナの幅制限を確認する。狭いカード（300px以下）内では overflow-wrap: anywhere を併用する。テキストカードのグリッドは最大3列。
32. **`.section-heading` の `align-items` は必ず `start` を使用する（`end` 禁止）**。`end` を使うと段落がh2より上に浮き上がるレイアウトバグが発生する。
33. **portrait / profile 画像には必ず `max-height: 420px` を設定する**。列幅が広いと縦に伸びすぎてバランスが崩れる。
34. **2カラムグリッド（company-grid / contact-grid 等）は `align-items: stretch` を使用する**。`start` では左右の下端が揃わない。
35. **全 `<img>` タグに `width` と `height` 属性を必ず設定する**（hero含む）。未設定は CLS（累積レイアウトシフト）の原因になり Lighthouse スコアを低下させる。
36. **`html` と `body` の両方に `overflow-x: hidden` を必ず設定する**。`body` だけでは Safari iOS でマーキー等の `width: max-content` 要素がページ幅を膨張させ、コンテンツが画面左半分に寄り右半分が空白になるバグが発生する。Chrome / Puppeteer では再現しないため実機確認が必須。
37. **CSS Grid の `grid-template-columns: 1fr` は `minmax(0, 1fr)` で書く**。`1fr` は `minmax(auto, 1fr)` の略であり、アイテムの min-content がコンテナ幅を超えるとトラックが膨張し水平オーバーフローを起こす。モバイルブレークポイント内・セクションラッパーの単一列定義は全て `minmax(0, 1fr)` にする。
38. **`display: grid` を持つセクションラッパーには `grid-template-columns` を必ず明示する**。省略すると暗黙の auto トラックがアイテムの max-content 幅で確定し、コンテンツが画面幅を超えてもクリップされない。`display: grid;` と書いたら必ず次行に `grid-template-columns: minmax(0, 1fr);` を書く。
39. **flex コンテナ内に `display: grid` を持つ要素は `min-width: 0` を追加する**。flex アイテムのデフォルト `min-width: auto` はアイテムの min-content で決まるため、内部の grid が想定より広い幅を要求するとコンテナが押し広げられる。

### オートレイアウト崩れ検査チェックリスト（生成後・デプロイ前）

- [ ] モバイル375px での `body.scrollWidth` が 375 以下か（Puppeteer または DevTools で確認）
- [ ] `display: grid` のある全要素に `grid-template-columns` が明示されているか
- [ ] モバイルブレークポイント内の `1fr` が `minmax(0, 1fr)` になっているか
- [ ] flex コンテナ内の grid 要素に `min-width: 0` があるか

## 5b. アニメーションルール（画像・要素別）

> 原則：**アニメーションの唯一の正当な理由は「視線誘導」**。
> 写真は静止している方が質・信頼感が高く見える。動かす理由がない要素には付けない。

### 画像へのアニメーション：原則禁止

以下の要素への `fade-in` / `translate` 付与を**禁止**する：

- `img` タグ全て
- `figure` / `.image-card` / `.gallery-item`
- `.service-card` / `.strength-card` / `.works-card` / `.gallery-card`
- `.profile-card` / `.portrait-stack` / `.member`
- `.stat-card`（カード全体）
- `.company-gallery` 内の全要素

**禁止理由**：
- `translateY` → グリッド内で隣接要素と視覚的に衝突する（今回発生した崩れの原因）
- 写真・作品は静止している方が重厚感・プロフェッショナル感が出る
- グリッド内で全カードが同時に動くと視線が散乱し、誘導効果がない

### アニメーションを付けて良い要素（限定許可）

| 要素 | 許可アニメーション | 備考 |
|------|-----------------|------|
| セクション見出し `h2` | `opacity` + `translateY(16px)` | 読む順番を作る |
| サブコピー・本文 `p` | `opacity` + `translateY(12px)` | h2より少し遅延させる |
| `section-kicker` | `opacity` のみ | |
| CTAボタン | hover時の `transform: scale` のみ | 既存ルール通り |
| ヒーロー内キャッチコピー | `opacity` + `translateY(20px)` | 最初の登場演出として有効 |
| ナビ・モーダルの開閉 | `opacity` + `transform` | UIフィードバック |

### 例外：単独画像の opacity フェード（条件付き許可）

以下を**全て**満たす場合のみ `opacity` フェード（translateY なし）を許可：
1. グリッド外の単独配置（1カラムで画面幅の50%以上を占める画像）
2. `opacity` のみ（`translateY` 禁止）
3. トランジション 0.8s 以内

### アニメーションが有効なシチュエーション（設計判断の参考）

- ✅ ページ初回表示時のヒーロー文字の登場
- ✅ スクロールで新しいセクションの見出しが現れる
- ✅ 統計数値セクションのカウントアップ
- ✅ フォームの送信完了メッセージの表示
- ✅ ハンバーガーメニューの開閉
- ❌ 既にスクロールして見える位置にある画像
- ❌ グリッド内の複数カードが一斉に動く
- ❌ 作品・料理・建築写真（静止している方が価値が伝わる）

- 文体レベル（A/B/C）が業態デフォルトに合っているか

- 禁止カタカナ（ソリューション/オプティマイズ等）が使われていないか
- 単語の途中や助詞の前で改行が発生していないか
- word-break: keep-all が見出し・コピーに設定されているか
- コンセプト文が具体性テストに合格するか（サロン名を隠しても特定できるか）
- 読点が適切か（20文字以上読点なし、1文に読点3個以上がないか）

## 16. インフラ・デプロイ品質

> 2026-04-14〜18 の siteon.jp SSL 障害（https_enforced リセット起因）を受けて追加。

### GitHub Pages 変更後の必須チェックリスト

Organization 移管・リポジトリ移動・Settings 変更・カスタムドメイン変更を行った後は、以下を毎回確認する：

1. **Custom Domain が Settings → Pages で設定されているか**
   - 確認コマンド：`gh api repos/{owner}/{repo}/pages --jq '.cname'`
   - 期待値：`"siteon.jp"` のような設定済みドメイン名

2. **CNAME ファイルがリポジトリルートに存在するか**
   - 確認コマンド：`gh api repos/{owner}/{repo}/contents/CNAME --jq '.content' | base64 -d`
   - 期待値：ドメイン名 1 行（改行のみ）

3. **Enforce HTTPS（https_enforced）が true になっているか** ⚠️ 最重要
   - 確認コマンド：`gh api repos/{owner}/{repo}/pages --jq '.https_enforced'`
   - 期待値：`true`
   - **Web UI では気づきにくい設定**。必ず API 経由で確認する
   - false のままだと GitHub は `*.github.io` 用ワイルドカード証明書を返し ERR_CERT_COMMON_NAME_INVALID が発生する

4. **HTTPS で 200 応答があるか**
   - 確認コマンド：`curl -sI https://{domain} | head -1`
   - 期待値：`HTTP/2 200`

5. **証明書の subject が正しいドメイン名か**
   - 確認コマンド：`echo | openssl s_client -servername {domain} -connect {domain}:443 2>/dev/null | openssl x509 -noout -subject`
   - 期待値：`subject=CN = {domain}`（例：`CN = siteon.jp`）
   - `CN = *.github.io` が返ってきたら https_enforced が false の可能性

### トリガー条件（このチェックをいつ実行するか）

以下の作業を行った場合、**直後に必ず全 5 項目を実行**する：

- Organization 移管（y1-310 → novatech-studio 等）
- リポジトリ名変更
- Settings → Pages の設定変更（Source ブランチ変更・Custom domain 追加/削除含む）
- CNAME ファイルの編集
- Xserver 等の DNS レコード変更
- カスタムドメインの新規追加

### 月次保守での定期実行

毎月 1 日に `tools/github-pages-health-check.js` を実行し、全クライアントサイトの状態を Slack `#alerts` に報告する。

### 異常検知時の対応手順

`https_enforced = false` が検出された場合の修復手順：

```bash
# Custom Domain 再設定（証明書プロビジョニングを再トリガー）
gh api -X PUT repos/{owner}/{repo}/pages -f cname="" && \
gh api -X PUT repos/{owner}/{repo}/pages -f cname={domain} -F https_enforced=true
```

修復後、10〜60 分待機して上記チェックリスト 5 項目で再確認する。

### 禁止事項

- Web UI 確認だけで「設定済み」と判断しない（https_enforced が見えにくい）
- Let's Encrypt 証明書再発行中（10〜60 分）に設定を何度も変更しない
- DNS 変更は必ず Yuichi に依頼（Xserver は手動操作必須）
